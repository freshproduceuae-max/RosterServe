RS-F005 ARCHITECTURAL DECISIONS (from approved plan)
- availability_blockouts is a separate table from availability_preferences (RS-F004) — additive, no alteration to existing tables
- Soft-delete pattern: deleted_at column; removeBlockout sets deleted_at = now(); permanent purge deferred to RS-F014
- Partial unique index (volunteer_id, date) WHERE deleted_at IS NULL — allows re-add of a previously soft-deleted date
- Two-step inline confirmation before soft-delete (satisfies confirmation guardrail without a full modal)
- Leader planning scope = volunteer_interests → departments (owner_id) for dept_head; volunteer_interests → departments → sub_teams (owner_id) for sub_leader — provisional, to be replaced by assignment-based scope in RS-F008
- New profiles RLS policy added in 00007: leaders can read in-scope volunteer profiles for display_name; additive alongside existing 00004 policy
- AppNav now accepts role prop; layout.tsx renders it for all roles; Events link gated by isLeaderRole()
- v1 is blockout-first: dated unavailability only; positive dated availability not modeled; general preferences remain in availability_preferences (RS-F004)
