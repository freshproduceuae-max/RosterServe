EVENT ARCHITECTURAL DECISIONS (RS-F002)
- Event type enum: Postgres enum event_type with 3 values (regular, ad_hoc, special_day)
- Event status enum: Postgres enum event_status with 3 values (draft, published, completed)
- Status transitions: enforced at database level via BEFORE UPDATE trigger. Valid: draft->published, draft->completed, published->completed. No backward transitions.
- Authorization model: super_admin creates/edits/deletes events. dept_head and sub_leader have read-only access to all active events (pre-RS-F003 baseline — visibility should be tightened to department-scoped when RS-F003 introduces ownership). Volunteers have no direct event access.
- Soft-delete: deleted_at column, same pattern as profiles. No hard DELETE via RLS.
- RLS policies: 5 policies (super_admin read all including deleted, leaders read active, super_admin insert, super_admin update)
- Server actions: createEvent, updateEvent, transitionEventStatus, softDeleteEvent — all with Zod validation and role checks
- URL structure: /events (list), /events/new (create), /events/[id] (detail), /events/[id]/edit (edit)
- Page-level authorization: /events and /events/[id] enforce isLeaderRole(); /events/new and /events/[id]/edit enforce hasMinimumRole(super_admin) with server-side redirect
- Navigation: AppNav client component with active state underline indicator
