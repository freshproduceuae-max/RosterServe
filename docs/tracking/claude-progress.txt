RS-F004 ARCHITECTURAL DECISIONS (from approved plan + Codex addendums)
- Gate location: (app)/layout.tsx adds one redirect for volunteer + !onboarding_complete → /onboarding; middleware stays thin
- Onboarding route group: (onboarding)/ with its own minimal layout — no nav bar, warm-surface wrapper; layout gate ejects non-volunteers and already-onboarded users back to /dashboard
- Three steps: (1) availability preferences — general days/times only, NOT dated blockouts (RS-F005); (2) department interests — FK to departments rows, skippable if none exist; (3) skills — free text, status=pending, approval deferred to RS-F007
- Step data saved incrementally per step (server action per step); partial data persists on browser close
- saveVolunteerSkills is APPEND-ONLY with dedupe — does not replace on re-submit (Codex addendum)
- getActiveDepartmentsForInterests and getOnboardingState must filter departments.deleted_at IS NULL at query layer — stale interest rows from soft-deleted depts are invisible at UI layer (Codex addendum)
- completeOnboarding() sets profiles.onboarding_complete = true then redirects to /dashboard; must be idempotent
- Migration 00005_onboarding.sql: three new tables; purely additive, no existing tables altered
- availability_preferences: UNIQUE on volunteer_id (one row per volunteer, upserted)
- volunteer_interests: UNIQUE (volunteer_id, department_id); no status column — that is RS-F006
- volunteer_skills: status column exists from day one (pending/approved/rejected) so RS-F007 can drive it without a migration
- Downstream contracts: RS-F005 extends availability with dated blockouts table; RS-F006 adds status/routing to interests; RS-F007 adds approval workflow UI on top of existing skills table
- Existing volunteers (onboarding_complete=false): gated on first sign-in after deployment — intentional
- Redirect-loop safety: (app)/layout.tsx gates incomplete volunteers out; (onboarding)/layout.tsx gates complete/non-volunteer back — no bounce (Codex addendum)
